## B-M4-1 Resolution ‚Äî Verified

**Original defect:** `afterAll` deleted rows using `WHERE "externalEventId" LIKE 'test-m4-%'`. The first Instagram test stores `externalEventId = 'ig_test-m4-ig-001_1700000001'` (generated by `extractInstagramEventId`'s `ig_${entryId}_${entryTime}` fallback). That ID starts with `ig_`, not `test-m4-`, so it survived cleanup. On the second run the row triggered the idempotency path (PENDING ‚Üí SKIPPED) and the test assertion `expect(event.status).toBe('PENDING')` failed.

**Fix applied** (`webhook.integration.test.ts` lines 65‚Äì83):

1. Extracted a `cleanTestData()` helper that deletes on both patterns:
   ```sql
   DELETE FROM webhook_events
   WHERE "externalEventId" LIKE 'test-m4-%'
      OR "externalEventId" LIKE 'ig_test-m4-%'
   ```

2. Added `beforeAll` that calls `cleanTestData()` before the suite starts ‚Äî defensive cleanup that removes any residue from a previously-interrupted run regardless of whether `afterAll` ran.

3. `afterAll` now calls the same `cleanTestData()` ‚Äî covers both pattern families.

4. `beforeAll` imported into vitest imports alongside `afterAll` (line 8).

**Scope confirmed:** All changes are in `tests/integration/webhook.integration.test.ts`. No production source files were touched. `grep` of `src/` shows no diff.
